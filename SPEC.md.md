# مواصفات نظام BRIN — منيو إلكتروني + كاشير + ولاء + إدارة

## نظرة عامة
مطعم برجر (BRIN) بفرع واحد، جلسات محلية صغيرة، 3 موظفين. نموذج استلام (Counter Service/Pickup) — بدون خدمة توصيل أو ربط طاولات.

## هوية بصرية
- الخلفية: `#F8F8F6`
- اللون الرئيسي: `#B82F36`

---

## المكدّس التقني (Tech Stack)

| الطبقة | التقنية | السبب |
|---|---|---|
| Frontend (3 مواقع) | Next.js + TypeScript + Tailwind CSS | SSR سريع، يُبنى كصورة Docker وينشر على Railway بسهولة، أمان الأنواع |
| Backend + DB | Supabase (PostgreSQL) | Realtime مدمج، Auth مدمج، Row Level Security |
| الاستضافة | Railway | نشر تلقائي من GitHub، 3 خدمات منفصلة (منيو/كاشير/إدارة) + قاعدة بيانات بمكان واحد إن احتجت، تحكم أكبر بالموارد والبيئة |
| الدفع الإلكتروني | Moyasar | مدى + Apple Pay + بطاقات، Webhooks، Refund API |
| OTP | Authentica | SMS، سعودي، قابل للتوسع لواتساب/نفاذ |
| الطباعة | Print Agent محلي (Node.js + ESC/POS) | يعمل على جهاز الكاشير، يطبع على طابعتين USB حراريتين |

## قنوات الطلب

### 1. طلب يدوي (كاشير)
- الموظف يدخل الطلب من واجهة الكاشير
- الدفع: كاش و/أو جهاز شبكة خارجي منفصل
- **دعم تقسيم الفاتورة** (كاش + شبكة بنفس الطلب، يجب أن يساوي مجموعهما الإجمالي بالضبط)
- تسجيل عميل جديد اختياري (اسم + جوال، بدون OTP)

### 2. طلب إلكتروني (المنيو)
- دفع أونلاين إجباري عبر Moyasar قبل وصول الطلب
- تسجيل دخول إجباري عبر Authentica (OTP بـ SMS)
- بعد نجاح الدفع (Webhook تأكيد) → إشعار Realtime فوري لشاشة الكاشير

### آلية استقبال الطلب الإلكتروني بالكاشير
- الشاشة الافتراضية = واجهة الطلب اليدوي
- طلب أونلاين جديد → **نافذة منبثقة + صوت تنبيه** (طابور لو عدة طلبات بنفس الوقت)
- **لا يوجد رفض** — فقط زر واحد: **"استلام وطباعة"** (لأن الدفع تم مسبقاً، والتوفر مضمون عبر تحديث المخزون اللحظي)
- إلغاء طلب بعد الاستلام = صلاحية **المدير فقط** من لوحة الإدارة (سبب إلزامي + استرجاع تلقائي عبر Moyasar API + تسجيل بـ Audit Log)

## المنيو

- فئات (تبويبات أفقية علوية)
- كل صنف: صورة + اسم + سعرات حرارية + سعر
- **تعديلات (Modifiers)**: إزالة (بدون كاتشب...) / إضافة (+جبن، +لحم بسعر) / اختيار إجباري (نوع الخبز)
- **وجبات (Combos)**: تجميع أصناف بسعر مخفض
- **قائمة مكافآت منفصلة** (Rewards Menu) يديرها المدير — نمط ماكدونالدز
- إغلاق تلقائي لاستقبال الطلبات خارج ساعات العمل (تُضبط من لوحة الإدارة)
- تحديث فوري لحالة "نفد المخزون" من لوحة الإدارة → يختفي الصنف فوراً بالمنيو (Realtime)

---

## نظام الولاء

- الاكتساب: **1 ريال = 10 نقاط** (تلقائي بكلا القناتين، بشرط وجود حساب عميل)
- الاستبدال: قائمة مكافآت محددة يختارها المدير، كل مكافأة لها تكلفة نقاط ثابتة
- الاستبدال متاح بكلا القناتين (منيو إلكتروني + كاشير)
- صفحة حساب العميل بالمنيو: رصيد النقاط، سجل الطلبات، إجمالي الإنفاق، تبويب المكافآت

---

## الطباعة

عند "استلام وطباعة" (يدوي أو إلكتروني) → طباعة تلقائية فورية على طابعتين USB حراريتين:
- **فاتورة مختصرة** (أصناف + كمية + ملاحظات، بدون أسعار) → طابعة المطبخ
- **فاتورة كاملة** (أصناف + أسعار + ضريبة + إجمالي + رقم طلب يومي) → طابعة العميل
- الفاتورة الكاملة تتضمن **QR متوافق مع ZATCA** (فاتورة ضريبية مبسطة)
- تنبيه واضح بالكاشير عند فقد الاتصال بالطابعة أو الإنترنت، إعادة محاولة تلقائية

## الموظفين والصلاحيات

- **مدير/مالك**: صلاحية كاملة (تقارير، أسعار، إعدادات، موظفين، إلغاء/استرجاع طلبات)
- **موظف**: تشغيل كامل (طلبات، طباعة، تسجيل عملاء) بدون تقارير مالية أو إعدادات حساسة
- دخول سريع بكود PIN (4 أرقام) لكل موظف، سجل دخول/خروج

## لوحة الإدارة

1. **الرئيسية**: مبيعات اليوم، عدد الطلبات، متوسط الطلب، مقارنة بالأمس، أعلى 5 أصناف، رسم بياني بالساعة
2. **إدارة المنتجات**: أصناف، فئات، تعديلات، وجبات، حالة التوفر
3. **إدارة الطلبات**: سجل كامل، فلترة حسب القناة/الحالة/التاريخ، إلغاء واسترجاع
4. **التقارير**: مبيعات (يومي/أسبوعي/شهري)، حسب طريقة الدفع، ضريبة، أداء الموظفين، **إقفال يومي (كاش متوقع مقابل فعلي)**
5. **CRM والولاء**: قاعدة العملاء، تعديل نقاط يدوي، إعدادات الاستبدال
6. **الموظفين**: إضافة حسابات، صلاحيات، Audit Log
7. **العروض والخصومات**: أكواد خصم، عروض بأوقات محددة
8. **الإعدادات العامة**: معلومات المطعم، الرقم الضريبي، ساعات العمل، QR الطاولات، إعدادات الطباعة

## البنية التقنية الكاملة (Architecture)

### نمط المستودع: Monorepo واحد

بدل 3 مستودعات منفصلة، نستخدم **مستودع واحد (Monorepo)** بأداة **Turborepo** — يسهّل مشاركة الأنواع (Types) والمكونات المشتركة بين المواقع الثلاثة، ويخلي التحديثات متزامنة بمكان واحد. مناسب جداً لمطور واحد أو فريق صغير.

```
brin-system/
├── apps/
│   ├── menu/                 # موقع المنيو الإلكتروني (عام)
│   ├── pos/                  # موقع الكاشير (محمي)
│   └── admin/                # لوحة الإدارة (محمي)
├── packages/
│   ├── database/             # عميل Supabase + الأنواع المولّدة من قاعدة البيانات
│   ├── ui/                   # مكونات واجهة مشتركة (أزرار، بطاقات، Modals)
│   ├── config/                # ثوابت مشتركة (ألوان الهوية، إعدادات عامة)
│   └── utils/                 # دوال مساعدة مشتركة (حساب الضريبة، تنسيق العملة)
├── print-agent/               # تطبيق Node.js منفصل يعمل محلياً على جهاز الكاشير
├── supabase/
│   ├── migrations/            # ملفات SQL لهيكلة قاعدة البيانات (مرقمة بالترتيب الزمني)
│   └── seed.sql                # بيانات تجريبية أولية
├── .env.example
├── .gitignore
├── turbo.json
└── package.json
```

### لماذا هذا الترتيب؟
- **فصل واضح لكل واجهة** (`apps/`) لكنها تتشارك المنطق المشترك (`packages/`) بدون تكرار كود
- تعديل بمكون مشترك (مثلاً بطاقة منتج) ينعكس تلقائياً بكل مكان يستخدمه
- `print-agent` منفصل تماماً لأنه لا يعمل كموقع ويب، بل تطبيق محلي مستقل يُشغَّل بجهاز الكاشير فقط

## هيكلة كل تطبيق (مثال: `apps/pos`)

```
apps/pos/
├── app/
│   ├── (auth)/
│   │   └── login/              # تسجيل دخول PIN
│   ├── orders/
│   │   ├── page.tsx             # واجهة الطلب اليدوي الرئيسية
│   │   └── incoming/            # منطق استقبال الطلبات الإلكترونية (Realtime)
│   ├── layout.tsx
│   └── api/
│       └── print/               # نقطة اتصال بـ Print Agent المحلي
├── components/
│   ├── OrderModal.tsx
│   ├── SplitPaymentDialog.tsx
│   └── ProductGrid.tsx
├── hooks/
│   ├── useRealtimeOrders.ts     # الاشتراك بقناة Supabase Realtime
│   └── usePinAuth.ts
├── lib/
│   └── supabaseClient.ts
└── package.json
```

**نفس النمط** يتكرر بـ `apps/menu` و `apps/admin` (فولدرات `app/`, `components/`, `hooks/`, `lib/` بكل واحد) — تناسق كامل يسهّل على أي مطور (أو Claude Code) يتنقل بين المشاريع بنفس المنطق.

### مبادئ Clean Code المتبعة
- **فصل الاهتمامات (Separation of Concerns)**: منطق الاتصال بقاعدة البيانات بـ `lib/`، منطق الحالة بـ `hooks/`، العرض فقط بـ `components/`
- **Single Responsibility**: كل مكون/دالة تسوي شيء واحد بوضوح (مثال: `useRealtimeOrders` مسؤول فقط عن الاشتراك بالطلبات، مو عن عرضها)
- **بدون تكرار (DRY)**: أي منطق يُستخدم بأكثر من تطبيق يُرفع لـ `packages/`
- **تسمية واضحة**: أسماء ملفات ومتغيرات تشرح نفسها بدون الحاجة لتعليقات إضافية (مثل `calculateLoyaltyPoints()` بدل `calc()`)
- **معالجة الأخطاء الموحدة**: طبقة واحدة لمعالجة أخطاء Supabase/Moyasar/Authentica بدل تكرار try/catch بكل مكان

## هيكلة قاعدة البيانات (الجداول الأساسية)

| الجدول | الوصف |
|---|---|
| `categories` | فئات المنيو |
| `products` | الأصناف (اسم، سعر، سعرات، صورة، فئة، حالة التوفر) |
| `modifier_groups` / `modifiers` | مجموعات وخيارات التعديل (إزالة/إضافة/اختيار إجباري) |
| `combos` / `combo_items` | الوجبات وربطها بالأصناف الداخلة فيها |
| `customers` | العملاء (اسم، جوال، تاريخ التسجيل) |
| `loyalty_points` | رصيد ومعاملات النقاط (اكتساب/استبدال) لكل عميل |
| `rewards` | قائمة المكافآت وتكلفة نقاطها |
| `orders` | الطلبات (رقم يومي، القناة، الحالة، الإجمالي، وقت الإنشاء) |
| `order_items` | أصناف كل طلب مع التعديلات المختارة |
| `payments` | سجل الدفع لكل طلب (يدعم أكثر من صف بنفس الطلب لتقسيم الفاتورة) |
| `employees` | الموظفين (اسم، PIN مشفّر، الصلاحية) |
| `audit_log` | سجل الأحداث الحساسة (إلغاء، استرجاع، تعديل سعر) |
| `restaurant_settings` | إعدادات عامة (ساعات العمل، الرقم الضريبي، إعدادات الولاء) |

كل جدول يُبنى بملف Migration منفصل ومرقّم بـ `supabase/migrations/` (مثال: `0001_create_products.sql`) — يسهّل تتبع تاريخ التغييرات ومراجعتها بـ Git.

## معايير الأمان بالكود

- **متغيرات البيئة فقط** لكل مفتاح حساس (Moyasar, Authentica, Supabase Service Role) — أبداً لا تُكتب بالكود مباشرة
- **Row Level Security (RLS)** مفعّل على كل جدول بـ Supabase من أول Migration، مو إضافة لاحقة
- **التحقق من التوقيع (Webhook Signature)** لكل Webhook قادم من Moyasar قبل الوثوق بمحتواه
- **Server Actions / API Routes فقط** للعمليات الحساسة (خصم نقاط، تأكيد دفع) — أبداً لا تنفَّذ من طرف المتصفح مباشرة

## سير العمل بـ Git

- فرع `main` = الإنتاج فقط (مربوط تلقائياً بالنشر على Railway)
- فرع `dev` = التطوير والتجربة
- كل ميزة جديدة بفرع منفصل (`feature/loyalty-redemption`) ثم Merge بعد المراجعة
- رسائل Commit واضحة بصيغة موحدة (مثال: `feat: add split payment to POS`, `fix: correct loyalty points calculation`)
